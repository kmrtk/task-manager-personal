# CLAUDE.md — Claude Code 運用ルール

## GitHub フロー（必ず守ること）

### 作業の流れ

1. **Issue を作成する** — 作業を始める前に必ず GitHub Issue を作る
2. **ブランチを切る** — Issue 番号を含むブランチ名で作業ブランチを作成する
3. **実装する** — ブランチ上でコミットを積む
4. **PR を作成する** — master へのマージは必ず Pull Request 経由で行う
5. **master への直接プッシュは禁止**

### ブランチ命名規則

```
feature/#{Issue番号}-短い説明   # 新機能
fix/#{Issue番号}-短い説明       # バグ修正
docs/#{Issue番号}-短い説明      # ドキュメント
```

例：
- `feature/#12-task-read-api`
- `fix/#7-folder-null-error`
- `docs/#3-api-spec`

### Issue の書き方

- **タイトル**: 何をするか一言で（例: `タスク読み取りAPIを実装する`）
- **本文**: なぜ必要か、何をするか、完了条件を書く

### コミットメッセージ

```
#{Issue番号} 変更内容の要約
```

例: `#12 Task エンティティとリポジトリを追加`

### PR の書き方

- タイトルに Issue 番号を含める（例: `#12 タスク読み取りAPI実装`）
- 本文に `Closes #12` を書くと Issue が自動クローズされる

## ポート管理ルール（必ず守ること）

このプロジェクトで使用するポートは以下の通り固定する。別ポートでの起動は禁止。

| サービス | ポート |
|---|---|
| nginx（フロントエンド） | 80 |
| Spring Boot（バックエンド） | 8080 |
| PostgreSQL | 5432 |

### ポート競合が発生した場合の対処

別のプロセスがポートを使用中でサービスが起動できない場合は、**そのプロセスを停止してから**指定ポートで起動する。別ポートへの変更は行わない。

```powershell
# 使用中のポートを確認（例：8080）
netstat -ano | findstr :8080

# PIDを確認してプロセスを停止
taskkill /PID {PID番号} /F
```

ポートを変更すると nginx のプロキシ設定や application.yml との整合性が崩れるため、
設定ファイルを書き換えてのポート変更は行わないこと。

## 破壊的な git 操作の禁止（必ず守ること）

以下のコマンドはローカルの変更を失う可能性があるため、**必ずユーザーに確認してから実行する**。

- `git clean -fd` — 未追跡ファイルを強制削除
- `git reset --hard` — コミット履歴ごと変更を破棄
- `git checkout -- .` / `git restore .` — 変更を破棄
- `git push --force` — リモートの履歴を上書き

## ブランチ切り替え前のルール（必ず守ること）

**背景:** 2026-06-17 の git clean -fd 事故を受けた対策。作業終了時に発覚したが、根本原因はブランチ切り替え前にコミットしていなかったことであり、作業の途中でも同様の事故が起きうるため追加。

`git checkout` / `git switch` でブランチを切り替える前に、以下を必ず実施する。

1. `git status` で未コミットの変更がないか確認する
2. 未コミットの変更があればコミット → プッシュを完了してから切り替える

---

## 作業終了時のルール（必ず守ること）

「今日はここまで」「終了」などの作業終了の指示があった場合、以下を必ず実施してから終了する。

1. `git status` でローカルに未コミットのファイルがないか確認する
2. 未コミットのファイルがあれば Issue を作成してブランチを切り、コミット＆PR を作成する
3. 作業中のものが何も残っていない状態にしてから終了する

**理由:** リモート（GitHub）に上がっていれば内容をすぐ把握できる。ローカルにだけ残っている状態は他の環境から確認できず、作業の継続に支障が出る可能性がある。

## Claude Code への指示

- 実装作業を始める前に、対応する Issue が存在するか確認すること
- Issue がなければ作成を提案すること
- ブランチは必ず上記の命名規則に従って作成すること
- master へ直接コミット・プッシュしないこと
- ポート競合が発生した場合は競合プロセスを停止し、必ず指定ポートで起動すること
- 破壊的な git 操作は必ずユーザーに確認してから実行すること
- ブランチを切り替える前に git status を確認し、未コミットの変更があればコミット＆プッシュを完了してから切り替えること
- 作業終了の指示があったら git status を確認し、未コミットのものがあればコミット＆PRしてから終了すること
